# KernelSU 权限绕过改造 - 坑点总结

## 问题背景

用户希望修改 KernelSU，使其允许**任何应用**执行 `su` 命令直接获得 root shell，而不经过任何权限验证。

---

## 一、调用 su 的完整函数调用链和检查链路

### 1.1 正常流程（未修改时）

```
用户执行: /system/bin/su
    │
    ├─→ shell 调用 faccessat("/system/bin/su", ...)
    │       └─→ ksu_hook_faccessat()
    │               ├─→ 检查 ksu_su_compat_enabled
    │               ├─→ 检查 ksu_is_allow_uid_for_current(uid)  ← 【拦截点1】
    │               │       └─→ __ksu_is_allow_uid()
    │               │               ├─→ forbid_system_uid()  ← 系统 UID 1001-1999 被禁止
    │               │               ├─→ is_uid_manager()     ← Manager APK 特殊处理
    │               │               └─→ Bitmap/Array 查询
    │               └─→ 路径替换: su → sh (如果权限检查通过)
    │
    ├─→ shell 调用 newfstatat("/system/bin/su", ...)
    │       └─→ ksu_hook_newfstatat()
    │               ├─→ 检查 ksu_su_compat_enabled
    │               ├─→ 检查 ksu_is_allow_uid_for_current(uid)  ← 【拦截点2】
    │               └─→ 路径替换: su → sh (如果权限检查通过)
    │
    └─→ shell 调用 execve("/system/bin/su", ...)
            └─→ ksu_hook_execve()
                    ├─→ 检查 ksu_su_compat_enabled
                    ├─→ ksu_handle_execve_sucompat()
                    │       ├─→ 检查 ksu_is_allow_uid_for_current(uid)  ← 【拦截点3】
                    │       ├─→ 路径替换: su → ksud
                    │       └─→ escape_with_root_profile() → 提权到 root
                    │               └─→ 执行 ksud
                    │
                    └─→ 【前提条件】: 进程的 tracepoint flag 必须被设置
                            └─→ 如果 flag 被清除，hook 不会被调用！
```

### 1.2 进程启动时的 tracepoint flag 管理

```
zygote fork 出新应用进程
    │
    └─→ 应用进程调用 setresuid(真实UID)
            └─→ ksu_hook_setresuid()
                    └─→ ksu_handle_setresuid()
                            ├─→ 检查 is_uid_manager()
                            ├─→ 检查 ksu_is_allow_uid_for_current(uid)
                            │       └─→ 如果返回 false:
                            │               └─→ ksu_clear_task_tracepoint_flag_if_needed()
                            │                       └─→ 清除 tracepoint flag  ← 【拦截点4】
                            │
                            └─→ ksu_handle_umount()
```

### 1.3 进程标记逻辑

```
ksu_mark_running_process_locked() 在以下时机被调用:
    ├─→ 模块初始化时
    ├─→ tracepoint 注册时 (syscall_regfunc kretprobe)
    └─→ tracepoint 注销时 (syscall_unregfunc kretprobe)
            └─→ 遍历所有进程:
                    ├─→ ksu_root_process (uid=0 && KSU域) → 设置 flag
                    ├─→ is_zygote → 设置 flag
                    ├─→ is_shell (uid=2000) → 设置 flag
                    ├─→ is_init (pid=1) → 设置 flag
                    ├─→ ksu_is_allow_uid(uid) → 设置 flag
                    └─→ 其他 → ksu_clear_task_tracepoint_flag()  ← 【拦截点5】
```

### 1.4 sys_enter tracepoint 处理

```
CPU 执行 syscall
    │
    └─→ sys_enter tracepoint 触发
            └─→ ksu_sys_enter_handler()
                    ├─→ 检查 compat 任务
                    ├─→ 检查 ksu_dispatcher_nr >= 0
                    ├─→ 检查 ksu_has_syscall_hook(id) → 判断是否需要 hook
                    │       └─→ 读取 syscall_hooks[id] 是否为 NULL
                    │
                    └─→ 如果需要 hook:
                            ├─→ orig_ax/x8 = syscallno
                            └─→ syscallno = ksu_dispatcher_nr
                                    └─→ 下一次 syscall 返回时进入 dispatcher
```

### 1.5 Dispatcher 分发

```
syscall 返回 (进入 dispatcher slot)
    │
    └─→ ksu_syscall_dispatcher()
            ├─→ 检查 syscallno == ksu_dispatcher_nr
            ├─→ 读取 orig_nr = orig_ax/x8
            ├─→ 读取 fn = syscall_hooks[orig_nr]
            └─→ 执行 fn(orig_nr, regs)
                    ├─→ __NR_faccessat → ksu_hook_faccessat()
                    ├─→ __NR_newfstatat → ksu_hook_newfstatat()
                    ├─→ __NR_execve → ksu_hook_execve()
                    └─→ __NR_setresuid → ksu_hook_setresuid()
```

## 二、遇到的坑点总结

### 坑点 1: faccessat/stat hooks 中的权限检查

**问题描述**: 只修改了 execve hook 中的权限检查，但忽略了 faccessat 和 newfstatat hooks 中的权限检查。

**影响**: 未授权应用执行 `su` 时，faccessat 路径替换不生效，导致 shell 认为 `/system/bin/su` 不存在。

**代码位置**: `kernel/feature/sucompat.c`

```c
// ksu_handle_faccessat()
if (!ksu_is_allow_uid_for_current(current_uid().val)) {
    return 0;  // ← 权限检查失败，不做路径替换
}

// ksu_handle_stat()
if (!ksu_is_allow_uid_for_current(current_uid().val)) {
    return 0;  // ← 权限检查失败，不做路径替换
}
```

**修复**: 注释掉这两处权限检查。

---

### 坑点 2: 路径重定向到 ksud 而非 sh

**问题描述**: 第一次修改只移除了权限检查，但没有修改路径重定向，导致提权后执行的是 `/data/adb/ksud` 而非直接执行 `sh`。

**影响**: 用户期望直接获得 root shell，但实际执行的是 ksud（虽然也能工作，但用户明确要求直接执行 sh）。

**代码位置**: `kernel/feature/sucompat.c`

```c
// 修改前:
*filename_user = ksud_user_path();  // 执行 ksud

// 修改后:
*filename_user = sh_user_path();  // 直接执行 sh，获得 root shell
```

**修复**: 将路径改为 `sh_user_path()`。

---

### 坑点 3: tp_marker 中清除未授权进程的 flag

**问题描述**: `ksu_mark_running_process_locked()` 会遍历所有进程，未授权的进程会被清除 tracepoint flag。

**影响**: 进程的 hook 标记被清除后，faccessat/execve hooks 不会被调用。

**代码位置**: `kernel/hook/tp_marker.c`

```c
void ksu_mark_running_process_locked(void)
{
    // ...
    if (ksu_root_process || is_zygote_process || is_shell || is_init || ksu_is_allow_uid(uid)) {
        ksu_set_task_tracepoint_flag(t);  // 授权进程设置 flag
    } else {
        ksu_clear_task_tracepoint_flag(t);  // ← 未授权进程清除 flag
    }
}
```

**修复**: 将 `ksu_clear_task_tracepoint_flag(t)` 改为 `ksu_set_task_tracepoint_flag(t)`。

---

### 坑点 4: setresuid hook 中清除 flag

**问题描述**: 当 zygote fork 出新应用进程后，进程调用 setresuid 设置自己的 UID。此时 `ksu_handle_setresuid` 会检查 UID 是否在 allowlist 中，不在则清除 tracepoint flag。

**影响**: 这是最关键的坑点！即使修改了其他所有地方，这个坑点不修复，hook 仍然不会被调用。

**代码位置**: `kernel/hook/setuid_hook.c`

```c
int ksu_handle_setresuid(uid_t old_uid, uid_t new_uid)
{
    // ...
    if (ksu_is_allow_uid_for_current(new_uid)) {
        ksu_set_task_tracepoint_flag(current);  // 有权限，设置 flag
    } else {
        ksu_clear_task_tracepoint_flag_if_needed(current);  // ← 清除 flag！
    }
    // ...
}
```

**修复**: 移除 else 分支中的清除逻辑，始终设置 flag。

## 六、注意事项

1. **System UID 限制**: `forbid_system_uid()` 禁止 UID 1000-1999（除了 1000 system），这在 `__ksu_is_allow_uid()` 中实现。但因为权限检查已注释，这个限制不会影响 su 执行。

2. **SELinux 上下文**: 提权后的进程 SELinux 上下文由 `escape_with_root_profile()` 中的 `setup_selinux()` 设置，默认为 `u:r:su:s0`。

3. **seccomp 限制**: `escape_with_root_profile()` 会调用 `disable_seccomp()` 移除 seccomp 限制。

4. **模块系统**: 修改后，模块系统的 `post-fs-data.sh`、`service.sh` 等仍然可以正常工作，因为这些由 ksud 处理。但用户选择不依赖 ksud，所以不会执行这些脚本。

5. **sulog 日志**: `ksu_sulog_capture_sucompat()` 仍然会记录日志，但不会阻止执行。
